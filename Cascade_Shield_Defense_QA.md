# Cascade Shield — Adversarial Defense Q&A

This document serves as the absolute, adversarial baseline truth of the Cascade Shield project (Phase 2). Every question is answered by tracing the claim directly to the code, logs, or dataset that generated it.

---

## A. Dataset Authenticity

### 1. Which datasets are actually real?
* **Power network:** Synthetic. It is the standardized IEEE 123-bus test feeder (an academic benchmark, not a real utility grid). (`data/123Bus/`)
* **Water network:** Synthetic/Academic benchmark.
* **Road network:** **REAL.** Extracted from OpenStreetMap (OSM) for Udupi (`data/02-Semiurban/raw/map.osm.pbf`).
* **ICS telemetry:** Synthetic. It is traffic generated by Mininet/SCADA simulators running over the IEEE node topologies. (`data/simulator/ics_events.json`)

### 2. Which parts are synthetic?
* Power edges: Real academic benchmark.
* Water edges: Real academic benchmark.
* Road edges: Real OSM.
* Cross-layer edges: **100% Synthetic.**
Total percentage synthetic: 100% of the cross-layer topology is generated algorithmically, representing roughly 5% of all edges.

### 3. Is the topology itself real?
**No.** The 2,278-node tri-layer graph is a synthetic fusion. The IEEE power nodes and water nodes were arbitrarily assigned (x, y) coordinates to fit within the geographic bounding box of the real Udupi road map.

### 4. How was graph fusion performed?
Nearest-neighbor Euclidean distance. `data/graph-engine/ingest.py` iterates through power/water nodes, calculates the Euclidean distance to all OSM road intersections, and adds a synthetic undirected edge to the nearest road node within a fixed radius. Ties are broken by array insertion order.

### 5. Why 1000m?
Arbitrary heuristic choice. Sensitivity to 500m vs 2000m was calculated but never formally trained or evaluated. 

---

## B. ICS Data

### 6. What exactly is "ICS telemetry"?
It is a JSON payload of synthetic IDS alerts (`ics_events.json`). It contains timestamps, node IDs, and event types (e.g., "Infected") simulating SCADA alarm logs.

### 7. Are PWR111–114 real substations?
**No.** They are arbitrary nodes within the IEEE 123-bus academic test feeder.

### 8. Only four origins?
Yes. We audited `ics_events.json` via a literal diff (`scratch/check_ics_cascade_profile.py`). The simulated IDS alerts only ever targeted `PWR_111`, `PWR_112`, `PWR_113`, and `PWR_114`. 

### 9. Is there train/test leakage?
**Macroscopically, yes.** While the *scenarios* are split 700/150/150, every scenario takes place on the exact same 2,278-node graph. Because the test set uses the same 4 origins as the training set on the same topology, the test set is a pure interpolation test, not an out-of-distribution generalization test.

---

## C. Simulation

### 10. Is the SEIR model physically realistic?
**No.** Propagation probabilities are hardcoded scalars in `simulator/discrete_event_sim.py` (e.g., PHYSICAL=0.65, LOGICAL=0.80). They are not calibrated to physics equations or epidemiological literature.

### 11. Can someone reproduce your simulation?
**Yes.** `data/simulator/generate_dataset.py` deterministically samples the ICS events using a fixed random seed.

### 12. Monte Carlo
One run per sampled scenario. The pipeline runs `discrete_event_sim.py` exactly once per sampled origin node. Convergence of the SEIR distribution was not tested.

---

## D. Model

### 13. Why GNN?
The project's premise was to test graph learning. However, as documented in the Phase 2 report, the GNN only marginally outperforms a zero-parameter Dijkstra heuristic (0.9089 vs 0.8991).

### 14. Which architecture?
3-layer GATv2Conv (Graph Attention Network v2) with residual connections. (`models/chokepoint.py`)

### 15. Number of parameters?
* Layers: 3
* Hidden size: 64
* Optimizer: Adam (lr=0.001)
* Batch size: 1 (Unbatched natively via `evaluate()`)
* Epochs: 100

### 16. Why Pairwise Ranking Loss?
It was implemented to counter the extreme 91% safe-node class imbalance. However, **it was disabled** (`lambda_rank=0.0`) during the verified Phase 2 training runs. The model trained purely on Focal Loss + MSE.

### 17. Why does hybrid fail?
The `train_hybrid` flag was never actively used in Phase 2. The edge-prediction head weights were effectively randomized, yielding the 0.53 AUC.

## E. Features

### 18. List every feature.
0. Criticality (randomized scalar)
1. Is Power (one-hot)
2. Is Water (one-hot)
3. Is Road (one-hot)
4. Is Transit (zeroed out)
5. In-degree (normalized)
6. Out-degree (normalized)
7. Is Compromised (binary origin marker)
8. Normalized Dijkstra Distance from Origin

### 19. Which features are synthetic?
All of them, but notably **Feature 8** is entirely calculated via the `networkx` shortest-path algorithm prior to training.

### 20. Which feature contributes most?
Feature 8.

### 21. Distance feature
As verified by the Ablation Run, removing Feature 8 causes the GNN to drop from 0.9089 to ~0.83-0.84 AUC. The other features remain included because message passing successfully uses them to *approximate* the missing distance signal.

## F. Evaluation

### 22. Dataset split
* 700 Train scenarios
* 150 Val scenarios
* 150 Test scenarios
Each graph has 2,278 nodes.

### 23. Why N=3?
Time and compute constraints (local CPU/Kaggle quotas limit 100-epoch runs to ~40 minutes each).

### 24. Confidence intervals
Mean ± std was reported for the N=3 baseline because it is standard shorthand, but formal 95% CIs were not constructed due to the small sample size.

### 25. Statistical significance
No formal paired t-test or Wilcoxon signed-rank test was performed between 0.9089 and 0.8991. 

### 26. Why ROC-AUC?
ROC-AUC was standard in Phase 1. For Phase 2, we heavily pivoted to **Precision@K (P@K)** (0.6790) as the primary metric, specifically to account for the heavy class imbalance.

# Round 2 — Deeper Interrogation (The "Real World" Assumptions)

## A. The "Real Dataset" Framing

### 1. Why do you repeatedly call this a "real dataset" or "real-world infrastructure" when over half the graph is synthetic?
The report's terminology has been aggressively corrected. This is **not** a real-world infrastructure graph. It is a **synthetic fusion** of three independent inputs:
* Real Geographic Data (OSM Roads)
* Academic Benchmark (IEEE 123-bus Power)
* Synthetic Simulation (Mininet/SEIR cascade logic)
The term "real-world" only legitimately applies to the geographic coordinates of the Udupi road intersections.

### 2. Random Criticality Feature
* **Is it fixed?** Yes. It is generated once during `generate_dataset.py` and saved permanently in the `.pt` file.
* **Does removing it change performance?** Unknown. We did not ablate Feature 0 alone. Including a randomized feature without ablating it is a known structural weakness, acting as arbitrary noise that the GNN may have overfit to.

### 3. Water Layer
* **Why include it?** The original project proposal aimed for a tri-layer graph. However, because the SEIR simulator currently only propagates along power edges (`simulator/propagation_rules.py`), the Water nodes act purely as static feature padding. They contribute no dynamic value to the cascade prediction.

### 4. Transit Layer
* **Why include an unused modality?** It is a deprecated artifact of Phase 1. It is currently zeroed out in the feature vector. It has no mathematical impact on the model, but remains in the codebase as technical debt.

### 5. Cross-layer edges
* **Why is Euclidean valid for infrastructure?** It is **not** physically valid. It is an engineering heuristic. Real utility routing follows road right-of-ways, topography, and underground corridors. Euclidean nearest-neighbor creates physically impossible connections (e.g., crossing through buildings or lakes).

### 6. Node coordinates
* **Could a different random placement change performance?** **Absolutely.** A different arbitrary geographic placement of the IEEE nodes would fundamentally alter which road nodes they connect to. Because cascade propagation depends entirely on shortest paths, changing the Euclidean placement would rewire the entire graph structure and completely alter the cascade distributions.

## B. The "ICS" Framing

### 7. Synthetic ICS
* **Is this actually telemetry or merely simulator output?** It is **simulator output**. Mininet and a Modbus simulator generate the traffic based on the SEIR logic. It is not scraped from a real physical SCADA firewall.

### 8. Ground Truth
* **How do you know which nodes are infected?** We don't. The labels are generated 100% by the SEIR simulator (`discrete_event_sim.py`). The model is learning to predict our own arbitrary simulator rules, not physical reality.

### 9. Label leakage
* **When is `is_compromised` assigned?** Before propagation. It acts as the input seed. Neighboring nodes *can* trivially infer proximity to the label. That is the explicit goal of the model: given a starting point, predict the spread. It is an input feature, not a leaked label.

## C. The Model

### 10. Distance Feature
* **Does the GNN really learn physics or simply recover Dijkstra?** The Ablation Run proves the GNN is largely recovering Dijkstra. When handed explicit distance, the GNN gets 0.9089 (barely beating the 0.8991 Dijkstra heuristic). When distance is withheld, the GNN drops to 0.84, forced to mechanically approximate proximity via message passing.

### 11. Why GATv2?
* It was chosen as a default modern PyG architecture. GraphSAGE, GCN, and GIN were not formally compared.

### 12. Hyperparameter search
* **Chosen manually.** No formal grid search was executed.

### 13. N = 3
* **How much variance exists over 10-50 seeds?** Unknown. However, the Phase 2 baseline variance (±0.0003) is mathematically constrained by the fact that only 4 origins exist. Running 50 seeds on 4 origins would just yield the exact same micro-variance.

### 14. P@K
* **Why K?** K is dynamically calculated per-scenario as the total number of actually infected nodes in that scenario. It answers: "Of the top K nodes you ranked highest, how many were actually in the true positive K?"

### 15. Threshold (Ablation Fairness)
* **Did you remove other features?** No. We only ablated Feature 8 because the distance heuristic explicitly relies *only* on distance. Ablating the remaining features would essentially leave the GNN with an empty graph.

## D. Compute & Constraints

### 16. CPU only?
* **Training takes:** ~40 minutes for 100 epochs on a standard quad-core CPU.
* **Evaluation takes:** < 5 seconds.
* **Dataset generation:** ~15 minutes.

### 17. Memory usage
* Easily fits within 8 GB of RAM. The entire 2,278-node dataset is a ~5MB `.pt` file.

### 18. Ablation fairness
* **Was the heuristic removed too?** The heuristic is a standalone baseline, not a component of the GNN. It was kept in the comparison table explicitly to show how badly the distance-deprived GNN (0.84) loses to the simple Dijkstra math (0.8991).

### 19. Static graph
* **Can topology change?** No. 

### 20. Realism
* **Why 0.65 / 0.80?** Completely arbitrary assumptions designed to ensure cascades propagate far enough to be mathematically interesting, rather than dying out immediately.

### 21. Calibration
* **Does predicted probability 0.9 mean 90% chance?** **No.** The calibration script (`calibration_analysis.py`) proved the Brier score is virtually identical to random guessing. The model is a pure ranking function; its absolute probability outputs are wildly uncalibrated.

### 22. Explainability
* **Attention visualization?** Not implemented. GATv2 attention weights could theoretically be extracted, but we do not do so.

### 23. Failure modes
* **When does the model fail?** Not formally investigated.

### 24. Scalability
* **What happens at 200,000 nodes?** GATv2 scales as O(|E|). It would train easily, but memory footprint would expand significantly due to the dense attention matrices.

### 25. Graph density
* **Average degree:** ~2.4 (5,535 edges / 2,278 nodes). It is an extremely sparse, planar-like graph.

### 26. Reproducibility
* **Why manual copy?** Expedience. It relies on the human not making a typo.

## E. The Meta Questions

### 27. Scientific contribution
By stripping away the simulation and dashboard, the core scientific contribution is exposing the **representational limits of GNNs on sparse infrastructure graphs**, proving that message passing often simply mechanically approximates distance heuristics, and that handing the model explicit distance renders graph convolution redundant.

### 28. Novelty
Exposing the representational ceiling of GNNs versus simple distance heuristics on cascading failures.

### 29. Biggest limitation
The entire performance evaluation relies on a single fixed, synthetic topology and a single set of arbitrary spatial cross-layer fusion coordinates.

### 30. One experiment that would most strengthen the paper
Running a formal sensitivity analysis on the topological fusion parameters.

## The Single Most Important Unanswered Question

> **How sensitive are your results to the arbitrary design choices?** (1000m vs 500m thresholds, different coordinate assignments, different random criticality, different propagation probabilities).

**Answer:** Highly sensitive. The reported 0.9089 ROC-AUC is definitively an artifact of this specific synthetic configuration. Because cascade dynamics are structurally bound by shortest-paths, changing the Euclidean coordinate placement or the 1000m cross-layer threshold would rewire the graph and entirely change the predictive baseline. A full sensitivity analysis across 5 different randomized topology generations would be the absolute requirement for Phase 3.

# Round 3 — Scientific Validity and Conclusions

### 1. How do you know your conclusions aren't specific to one synthetic graph?
We do not. The conclusions drawn in this report are formally constrained to the specific synthetic fusion coordinates, the 1000m threshold, and the SEIR propagation constants selected. Running the 5-graph topological sensitivity analysis is the required next step to prove these properties generalize across arbitrary spatial configurations.

### 2. Is your GNN learning anything beyond shortest paths?
Yes, but only marginally (+0.0098 AUC). We did not perform a feature attribution analysis. The observed improvement may arise from the combination of node attributes and message passing, but this was not experimentally verified.

### 3. Why use machine learning at all?
Given that Dijkstra achieves 0.8991 and requires no training, deploying the GNN (0.9089) for this specific static topology is difficult to justify. The true value of the ML approach lies in **extensibility**. A distance heuristic is blind to node state; a GNN can theoretically ingest dynamic live sensor data (current flow, line temperature, valve pressure), allowing it to predict cascade propagation in domains where pure structural proximity fails.

### 4. Are you evaluating prediction or simulator replication?
**Simulator replication.** The ground truth labels come entirely from the SEIR simulator. The evaluation measures how accurately the GNN reproduces the behavior of the implemented simulator. It does not establish predictive validity for real infrastructure failures.

### 5. What exactly is the scientific hypothesis?
*We hypothesized that message passing could learn cascade structure beyond shortest-path distance. Our experiments show that explicit shortest-path distance dominates performance, while message passing partially reconstructs that information when distance is withheld.*

### 6. Where is uncertainty quantified?
It is not formally quantified. The 0.0098 AUC improvement over the heuristic has not been subjected to a formal statistical significance test (e.g., paired t-test). While the N=3 baseline standard deviation is extremely tight (±0.0003), we acknowledge the limitation that this improvement could theoretically reflect random variation.

### 7. Can another implementation reproduce your conclusions?
If another researcher rebuilt the pipeline based solely on our methodology (without our exact random seeds), their absolute AUC numbers would likely differ because the spatial fusion algorithm would generate different synthetic cross-layer edges. Based on the experiments presented here, distance dominance and message passing as structural recovery are supported for this synthetic benchmark. Whether the same relationship holds across independently generated topologies or alternative propagation models remains an open question.

### 8. What evidence supports your main conclusion?
The conclusion that distance dominates is supported by a perfectly interlocking set of four experiments:
1. Dijkstra shortest-path heuristic reaches 0.8991 AUC.
2. The full GNN only improves marginally to 0.9089 AUC.
3. Removing distance (Feature 8) drops GNN performance to ~0.83–0.84 AUC.
4. A distance-free MLP without message passing collapses to ~0.56 AUC.
*Therefore, distance is the dominant predictor, while message passing successfully recovers a substantial portion of that structural information when it is withheld.*

### 9. What would invalidate your conclusion?
If the same conclusion (distance dominance + message passing recovery) did not hold across multiple independently generated fused graphs, or under alternative physically-driven propagation models (e.g., actual DC power flow simulations rather than SEIR), then our conclusion about distance dominance would not generalize beyond this single synthetic graph.

### 10. What is the real contribution?
The ultimate scientific contribution of Cascade Shield is not the dashboard, the simulator, or the specific GATv2 architecture. It is this:

> **A rigorous empirical analysis showing that, on this synthetic infrastructure cascade benchmark, explicit shortest-path distance explains nearly all predictive performance, while graph neural network message passing primarily reconstructs that structural information when it is withheld.**
